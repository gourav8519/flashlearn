# FlashLearn

A full-stack spaced-repetition learning platform built with Next.js 16. Users turn notes into smart flashcards (optionally generated by AI), and the app schedules reviews using the SM-2 algorithm so learners remember more in less time.

**Live demo:** https://flashlearn-xi.vercel.app
**Repo:** https://github.com/gourav8519/flashlearn

---

## Why this, not a CRUD todo list

The assignment explicitly asked for something beyond basic CRUD. FlashLearn sits on three non-trivial mechanics:

1. **SM-2 spaced-repetition scheduler** — every review updates each card's `interval`, `ease`, `reps`, and `due` timestamp so the next appearance is computed, not manual.
2. **AI flashcard generation** — paste notes, Groq's LLM returns structured front/back pairs validated before persistence.
3. **XP + streak + leveling system** — review activity drives gamification that's stored server-side and survives logout.

## Features

- **Auth**: email/password signup + login, optional Google OAuth, JWT sessions, per-user data isolation on every API route.
- **Decks & cards CRUD**: create/update/delete decks and cards, all owner-scoped.
- **Study mode**: review cards with 4-button rating (Again/Hard/Good/Easy), live SM-2 rescheduling, XP awarded on each review.
- **AI generator**: users provide their own Groq API key (stored on their user doc and never returned to the client — only a masked `gsk_1234•••••abcd` preview is exposed); we never ship a shared key.
- **Streaks**: consecutive-day tracking with auto-reset on missed days.
- **Dashboard**: XP bar, streak card, due-card queue, 7-day activity sparkline, top decks.
- **Landing page**: hero, feature bento, how-it-works, science section (Ebbinghaus/SM-2), FAQ, CTA.

## Local setup

### 1. Prerequisites

- Node.js 20.9+ (tested on 25.x)
- MongoDB running locally, or a cloud MongoDB URI

### 2. Install

```bash
git clone https://github.com/gourav8519/flashlearn.git
cd flashlearn
npm install
```

### 3. Environment variables

Create `.env.local`:

```bash
MONGODB_URI=mongodb://localhost:27017/flashlearn

# NextAuth
AUTH_SECRET=generate-with-openssl-rand-base64-32
AUTH_TRUST_HOST=true
NEXTAUTH_URL=http://localhost:3000

# Google OAuth — leave "placeholder" to hide the Google button
GOOGLE_CLIENT_ID=placeholder
GOOGLE_CLIENT_SECRET=placeholder

# Groq — leave "placeholder" to force users to provide their own key in Settings
GROQ_API_KEY=placeholder
```

Generate `AUTH_SECRET` with `openssl rand -base64 32`.

### 4. Run

```bash
npm run dev
```

Open http://localhost:3000.

### 5. Build

```bash
npm run build
npm run start
```

### 6. Tests

```bash
npm run typecheck   # TypeScript --noEmit (also runs in CI)
npm run test:e2e    # Playwright e2e (starts its own dev server on :3100)
```

E2E tests create throwaway users with timestamped emails against your local MongoDB. To keep the test DB clean, point `MONGODB_URI` at a separate database before running them.

## API surface

All mutations require an authenticated session cookie.

| Method | Path                           | Purpose                                  |
| ------ | ------------------------------ | ---------------------------------------- |
| POST   | `/api/auth/signup`             | Create account                           |
| \*     | `/api/auth/[...nextauth]`      | NextAuth handlers (signin/callback/out)  |
| GET    | `/api/me`                      | Hydrate session — user, decks, cards, reviews |
| PATCH  | `/api/profile`                 | Update name/email/Groq API key           |
| GET/POST | `/api/decks`                 | List / create deck                       |
| GET/PATCH/DELETE | `/api/decks/[id]`    | Fetch / update / delete deck             |
| POST   | `/api/decks/[id]/cards`        | Add card to deck                         |
| PATCH/DELETE | `/api/cards/[id]`        | Update / delete card                     |
| POST   | `/api/reviews`                 | Record a review (SM-2 reschedule + XP)   |
| POST   | `/api/ai/generate`             | Groq-powered flashcard generation        |

## Security

- **Passwords** hashed with bcrypt (cost 10) — never stored or logged in plaintext.
- **Sessions** use HTTP-only JWT cookies signed with `AUTH_SECRET`.
- **Authorization** — every route that touches user data goes through `requireUser()` which returns 401 if there's no session, and all DB queries filter by that user's `_id`.
- **Input validation** — API route handlers validate payload shapes and reject malformed input with 400.
- **Groq API key** is per-user, stored on their user doc, never exposed to the client (only a masked preview is returned).

## Design decisions

- **MongoDB over Postgres** — the entity relationships (user → decks → cards, reviews referencing cards) are shallow and document-friendly, and Mongoose's schema validation gave fast iteration.
- **NextAuth JWT sessions** — avoids a session table, scales horizontally without sticky sessions.
- **SM-2** — public-domain algorithm with decades of validation. Each card carries its own state, so there's no cron job — scheduling is lazy and computed on review.
- **User-supplied Groq keys** — avoids a shared-key cost liability and makes the app deployable by anyone with a free Groq account.
- **Turbopack (default in Next 16)** — faster dev + build than webpack.
- **React Context over Redux** — app state is moderate (one user session, their decks/cards/reviews) and largely server-owned. A single `AppProvider` in [lib/app-context.tsx](lib/app-context.tsx) wraps the app and exposes `useApp()` with both state and mutation actions. Redux would add ceremony (actions, reducers, slices, devtools) without a real payoff here. For local UI state, plain `useState` is used. If the app grew to multi-tenant workspaces or realtime multi-user state, a dedicated store (Zustand / Redux Toolkit) would make sense.
- **Zod for request validation** — every API route parses its body through a schema before touching the DB. This centralises validation, gives readable error messages, and keeps route handlers focused on business logic. See [lib/validation.ts](lib/validation.ts).
- **Centralised error handler** — `withErrorHandling` in [lib/api-helpers.ts](lib/api-helpers.ts) wraps every route and maps Zod errors → 400, Mongoose `CastError` → 400 ("invalid id"), and unhandled exceptions → 500 without leaking internals.

## Known limitations

- **Rate limiting is in-memory** — the limiter in [lib/rate-limit.ts](lib/rate-limit.ts) stores buckets in a process-local `Map`. On Vercel's serverless runtime each function instance has its own memory, so limits are enforced per-instance, not globally. Under high concurrency an attacker can partially bypass the cap by hitting the platform from different regions. The correct production fix is a shared store (Upstash Redis, Cloudflare KV). For the assignment this was a conscious trade-off — it covers the common-case abuse vectors without introducing an external service dependency.
- **JWT session invalidation** — changing a password does not revoke existing sessions. JWTs are self-contained and can't be centrally expired without a DB-backed session table or a token-version field we'd have to stamp into every JWT. The current UX treats this as acceptable; a production build would add a `tokenVersion` to the User model and check it in the NextAuth `session` callback.
- **Dashboard hydration loads all user data up-front** — `/api/me` returns every deck, card, and recent reviews in one response. Fine for typical user sizes (tens of decks, hundreds of cards) but wouldn't scale to power users with thousands of cards. The fix would be lazy-loading cards per deck and reviews per-page.
